Open Architecture · Claude Code

The Agency Brain

A self-learning Claude Code system with persistent memory, session learning loops, and automated GitHub backup. Built from markdown files, two Node.js scripts, and a hook — no plugins required.

Build It Yourself — Folder Guide →

The Problem Claude Code Has By Default

And the three-layer system that solves it without any external tools.

Claude Code starts fresh every session. No memory of previous work, past decisions, or what broke last time. For a one-off task that's fine. For an ongoing project — an agency, a product, a business — it means re-briefing Claude every single session and watching it repeat the same mistakes. This system fixes that with three layers.

LAYER 01 — CONTEXT

Persistent Brain Files

Markdown files in brain/ are read at the start of every session. They give Claude your business context, current project state, and working rules before any task begins. No briefing required.

LAYER 02 — LEARNING

Session Learning Loop

Mistakes go into lessons.md. Wins go into patterns.md. A hook injects the most recent entries at session start automatically — the same mistake doesn't happen twice.

LAYER 03 — BACKUP

Auto Git Backup

A SessionEnd hook commits and pushes every changed file to GitHub after every session closes. No manual step. No lost work. The system backs itself up every time you stop working.

The Memory Learning Loop

Every session feeds the next one. This is how the system compounds over time.

Session Start
Claude Code launches. .claude/settings.json fires the SessionStart hook.
brain-bridge.cjs runs
Reads brain/lessons.md and brain/patterns.md, extracts the 5 most recent entries of each, outputs them into session context. Uses mtime caching — only re-parses if files changed.
Claude reads brain files
Startup protocol in CLAUDE.md instructs Claude to read brain/HOME.md, status-board.md, and the most recent session log before doing anything.
Work session happens
Claude works with full context of past mistakes, working patterns, current project state, and business rules. No briefing needed.
/session-log runs
Slash command creates a dated handoff file in docs/session-logs/, updates plan.md checkboxes, updates status-board.md, and proposes new lessons and patterns.
Lessons + patterns confirmed
Reviewed proposals get appended to brain/lessons.md (failures → rules) and brain/patterns.md (wins → repeat). These will be injected at the next session start.
Session ends
SessionEnd hook fires auto-git-backup.cjs. Stages all changed files (excludes .env), commits with a timestamp, pushes to GitHub. Exits 0 regardless of outcome — never blocks close.
Next session: smarter
brain-bridge.cjs injects the updated lessons and patterns. Claude applies them before the first task. The system compounds — each session builds on the last.

Why This Works

H
Hooks do the heavy lifting. brain-bridge.cjs and auto-git-backup.cjs run without Claude doing anything. The system is automated — not dependent on Claude following instructions.
M
Memory is just markdown. lessons.md and patterns.md are plain text. No database, no vector store, no plugin. They're read by a 60-line script and injected as context.
B
Backup is zero-touch. auto-git-backup.cjs runs on every session close. You never need to remember to commit. The only way to lose work is if git itself fails.
L
Learning is explicit, not magic. You review proposed lessons before they're added. The system learns what you confirm — not hallucinated patterns.

What it is

The Persistent Memory Cortex

A folder of markdown files that Claude reads at the start of every session to orient itself. The brain folder is the continuity layer — it gives Claude your business context, current project state, past mistakes, and working patterns before any task begins.

Why it matters

Without brain files, every session starts cold. Claude doesn't know what project you're working on, what decisions were made last week, or that a specific approach failed three sessions ago. Brain files eliminate re-briefing entirely. They're read once per session, take ~5 seconds to load, and give Claude a month of context in one pass.

Files inside
FilePurpose
HOME.mdMaster orientation. Business/project summary, startup protocol, directory map, task routing. First file read every session.
lessons.mdRunning log of session mistakes → rules that would prevent them. Auto-injected at session start via brain-bridge.cjs. Format: date header, what happened, rule, target file.
patterns.mdLibrary of approaches that worked unusually well. Counterpart to lessons.md. Tells Claude what to repeat, not just what to avoid.
status-board.mdLive project status. What's active, what phase each project is in, last completed action, next action, blockers. Updated at session end.
routing-table.mdTask type → files to load lookup table. "If working on X, load Y first." Prevents cold starts on any task type.
MEMORY.mdTool preferences, key decisions, workspace health notes. Updated immediately when a new tool or MCP is added — not at session end.
[name].mdFounder or team member profile. Voice, beliefs, background, sales style. @imported so it's available for all content writing without manual loading.

  3. Create brain/lessons.md with a header and empty entries section (add real entries after your first session)
  4. Create brain/patterns.md with the same structure as lessons.md
  5. Create brain/status-board.md with your current projects
  6. Create brain/routing-table.md as a task-type lookup table
  7. Create brain/MEMORY.md to track tool decisions
  8. Add a reference to HOME.md as step 1 of your CLAUDE.md startup protocol

What it is

The Command Layer

A markdown file at the project root that Claude Code reads automatically at session start. Contains behavioral rules, @import declarations that load other rule files, the startup protocol, and non-negotiables for the project.

Why it matters

CLAUDE.md is how you encode what Claude must do and how it must think for your project. Without it, Claude applies default behavior. With it, Claude knows your rules, your tools, and your guardrails before touching anything. The @import syntax loads other markdown files into every session — so you can break large rule sets into focused files without losing them.

Brain Map / Startup protocolNumbered steps Claude follows at the start of every session. Must be identical to brain/HOME.md startup steps.
@import declarationsLoads rule files into every session: @.claude/rules/brand-voice.md, @brain/[name].md, etc.
Behavioral rules"Do what was asked; nothing more" / "Never create files unless absolutely necessary" / "Always read before editing"
Planning ruleFor tasks with 3+ distinct steps: write the plan first, wait for confirmation before executing.
Security rulesNever hardcode API keys. Credentials at .env. Never commit .env.
Concurrency ruleAll related file reads/writes must be batched in one message — maximizes parallel tool use.
How to build it
  1. Create CLAUDE.md at your project root (not in a subfolder)
  2. Write the startup protocol section first — this is what Claude does every session
  3. Add @import lines for any rule file you want loaded automatically (brand voice, workflow rules, founder profile)
  4. Keep it under ~120 lines — use @imports to avoid bloat
  5. Important: the startup sequence in CLAUDE.md, brain/HOME.md, and .claude/rules/workflow.md must stay in sync — update all three whenever you change startup steps

- NEVER create files unless absolutely necessary

- NEVER save working files to the root folder

For any task with 3 or more distinct steps: write a numbered plan first (max 10 words
per step), wait for confirmation before executing. Single-step tasks skip the plan.

## Security Rules

- NEVER hardcode API keys or secrets
- Credentials live at `[PATH TO .env]`
- NEVER commit .env files

## Concurrency Rule

All related operations in one message. Batch all file reads, writes, and Bash
commands in parallel. Never run related operations sequentially when they can be parallel.

What it is

The Automation Layer

A folder at the project root containing settings, rule files, hook scripts, and slash command definitions. This is where the automation lives — the hooks that run automatically, the rules that load silently, and the commands that do complex multi-step work in one word.

Why it matters

Without .claude/, every session requires manual setup. With it, context is injected before Claude types a word, the git backup runs without you thinking about it, and a 15-step session-close process happens with one command. It's the difference between a system and a collection of instructions.

Sub-folders
PathPurpose
.claude/settings.jsonHook registrations. Connects SessionStart and SessionEnd events to Node.js scripts. This file makes automation happen.
.claude/rules/Markdown rule files loaded via @import in CLAUDE.md. Keep brand voice, workflow rules, and writing style here — not inside CLAUDE.md itself.
.claude/helpers/Node.js scripts that run as hooks. brain-bridge.cjs (SessionStart), auto-git-backup.cjs (SessionEnd), scaffold-client.cjs (on-demand).
.claude/commands/Slash command definitions. Create session-log.md here to define the /session-log command. Each file = one command.
.claude/agents/Sub-agent configuration files. Optional — define specialized agents for complex recurring tasks.
How to build it
  1. Create the .claude/ folder at project root
  2. Create .claude/settings.json with SessionStart and SessionEnd hooks (template below)
  3. Create .claude/helpers/brain-bridge.cjs — reads brain files, outputs recent lessons/patterns to session context
  4. Create .claude/helpers/auto-git-backup.cjs — stages, commits, and pushes at session end
  5. Create .claude/rules/ and add your rule markdown files
  6. Create .claude/commands/session-log.md to define the /session-log command
  7. Reference rule files in CLAUDE.md using @.claude/rules/[file].md syntax

Prompt — brain-bridge.cjs (SessionStart hook)
Create a Node.js CommonJS script at .claude/helpers/brain-bridge.cjs that:

1. Reads brain/lessons.md and brain/patterns.md from the project root
   (project root = two directories up from __dirname)

2. Extracts entries matching ### YYYY-MM-DD date headers using regex:
   /^### \d{4}-\d{2}-\d{2}/
   Never filter by footer instruction text — only date headers count as real entries.

3. Takes the 5 most recent lessons and 3 most recent patterns

4. Outputs them to stdout as formatted session context using console.log.
   Format the output so it reads as a clear "Context injected at session start" block.

5. Uses mtime-based caching: store mtime + parsed entries in
   .claude/helpers/.brain-cache.json. Only re-parse if file mtime changed.

6. Also checks ~/.claude/hooks/brain-reminder.txt — if the file exists and
   has content, log it as a reminder, then leave deletion to the human.

7. Always exits with process.exitCode = 0 — never throws, never blocks session start.
   Wrap everything in try/catch and log errors to stderr only.
Prompt — auto-git-backup.cjs (SessionEnd hook)
Create a Node.js CommonJS script at .claude/helpers/auto-git-backup.cjs that:

1. Runs git --version to check git is available. If not, skip and exit 0.
2. Runs git remote -v. If no remote configured, log "No remote — skipping" and exit 0.
3. Runs git status --porcelain. If clean, log "Nothing to commit" and exit 0.
4. Gets list of changed files. FILTERS OUT any file containing ".env" in the path.
   This is a security guard — never commit .env files.
5. Stages each remaining file individually with: git add -- "[file]"
6. Runs git diff --cached --name-only to confirm something is staged.
7. Commits with message: "session backup YYYY-MM-DD HH:MM"
   (use new Date().toISOString().slice(0,10) and new Date().toTimeString().slice(0,5))
8. Runs git push.
9. Catches all errors, writes them to .claude/helpers/git-backup-error.log,
   and logs a single-line summary to console. Never throws.
10. Always exits with process.exitCode = 0 — never blocks session close.
What it is

Session Memory and Decision Log

A docs/ folder containing dated session handoff files and a running decisions log. Session logs are the mechanism that lets Claude pick up exactly where the last session left off — without you re-explaining anything.

Why it matters

Without session logs, each session starts cold. You have to re-explain what was done, what decisions were made, and what comes next. With session logs, Claude reads the most recent one as part of its startup protocol and has complete context on current state, blockers, and next actions. The decisions log prevents re-litigating choices that were already made and reasoned through.

docs/session-logs/YYYY-MM-DD.mdOne file per session. What was worked on, what was completed, what was not finished, decisions made and why, errors and fixes, what next session starts with, brain map changes, proposed lessons and patterns.
docs/decisions.mdArchitecture and strategy decisions with trade-off reasoning. Updated when a significant choice is made that isn't obvious from the code or current state.
docs/research/[topic].mdResearch notes for specific topics. Created when doing deep research that should be referenced later.
How to build it
  1. Create docs/ at project root
  2. Create docs/session-logs/ subfolder

  5. Add "Read most recent docs/session-logs/ file" as step 7-8 of your startup protocol in CLAUDE.md and brain/HOME.md
  6. At the end of every session, run /session-log — the command creates and populates the log automatically

What it is

Standard Operating Procedures

A folder of step-by-step process documents for repeatable tasks. SOPs encode process knowledge so Claude doesn't have to re-derive steps each time, and so the process is the same every time it runs.

Why it matters

Without SOPs, complex repeatable processes get re-invented each time — and with subtle variations. Onboarding a new client, connecting a new MCP server, setting up GitHub access — these are multi-step processes with decision points that should run the same way every time. SOPs give Claude a complete checklist, not just a general direction.

How to build it
  1. Create sops/ at project root
  2. Create sops/index.md listing all SOPs with a one-line description each
  3. Create one SOP file per repeatable process — keep each under 50 lines
  4. Reference relevant SOPs in brain/routing-table.md under the appropriate task types
  5. When a SOP is created, immediately update sops/index.md and brain/routing-table.md

What it is

Internal Tooling

A tools/ folder for internal utilities — typically a local web dashboard that provides a visual interface to your project's data without needing the terminal. Optional but useful for teams or long-running projects.

Why it matters

A local command center dashboard lets you see task status, memory contents, session log history, and system health at a glance. It's particularly useful when working with non-technical collaborators who don't use the terminal, or for quickly reviewing what Claude did in the last session without opening files manually.

What it is

Client Project Folders

A clients/ folder where each client gets their own isolated project directory with its own CLAUDE.md, prd.md, plan.md, and docs/. Claude never mixes client contexts because each project folder is independently configured and loaded.

Why it matters

Client isolation is non-negotiable. Without it, Claude might apply rules from Client A to Client B, or mix up project phases, or use the wrong brand voice. Each client folder is a complete mini-project with its own startup rules. The scaffold script ensures every client folder has the same structure — consistently, in under 5 seconds.

Structure per client
PathPurpose
clients/[slug]/CLAUDE.mdClient-specific rules: trade, city, services, tone, platform, special requirements. Loaded when working on this client.
clients/[slug]/prd.mdWhat success looks like for this client. Their goals, current state, KPIs.
clients/[slug]/plan.mdPhase-based task list. Track progress with [x] checkboxes.
clients/[slug]/docs/session-logs/Per-client session logs. Separate from the main project session logs.
clients/[slug]/sitemap.jsonURL inventory for website clients. Used for planning and ClickUp task creation.
How to build it
  1. Create the scaffold script (scaffold-client.cjs) in .claude/helpers/ using the prompt below
  2. When onboarding a new client, run: node .claude/helpers/scaffold-client.cjs "[slug]" "[type]" "[city]" "[offer]"
  3. Fill in the generated CLAUDE.md, prd.md, and plan.md with client specifics
  4. Add the client to brain/routing-table.md and brain/status-board.md immediately
  5. Add the client row to brain/HOME.md active projects table
Prompt — scaffold-client.cjs
Create a Node.js CommonJS script at .claude/helpers/scaffold-client.cjs.

The script takes 4 CLI arguments:
  argv[2] = slug (URL-safe identifier, e.g. "smith-hvac")
  argv[3] = type (e.g., "HVAC", "Plumbing", "Roofing")
  argv[4] = city (e.g., "Austin")
  argv[5] = offer (e.g., "swarm-system", "swarm-domination")

It should create clients/[slug]/ with these files:
  CLAUDE.md — client-specific rules with placeholders for trade, city, platform, services
  prd.md — client success criteria template with sections: Overview, Goals, Current State, KPIs
  plan.md — Phase 1 task list template with example tasks
  docs/session-logs/.gitkeep — creates the folder
  docs/decisions.md — empty decisions log with header
  sitemap.json — empty JSON array []
  README.md — one-pager: client name, trade, city, offer, start date, key contacts (all placeholder)

Use fs.mkdirSync with { recursive: true } and fs.writeFileSync.
After creating files, console.log a summary of what was created.
Do not create the clients/ root folder — assume it already exists.

Also print a reminder: "Next: update brain/status-board.md and brain/routing-table.md with [slug]"

Pre-flight install ~5 min, before session

Run during the call's intro so attendees aren't waiting on npm install mid-demo. Two paths:

Full Claude Code install (recommended)

One-liner. Installs Node, Claude Code, GSD, clones the repo, runs setup.sh, registers the supervisor + 3 bridges, opens welcome.html.

macOS — Terminal:

/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/search-atlas-group/amm-toolkit/main/Scripts/quickstart-mac.sh)"

Windows — PowerShell as Administrator:

irm https://raw.githubusercontent.com/search-atlas-group/amm-toolkit/main/Scripts/quickstart-windows.ps1 | iex

MCP-only install (Claude Desktop / Cursor / Windsurf users)

curl -fsSL https://raw.githubusercontent.com/search-atlas-group/amm-toolkit/main/Scripts/install-mcp.sh | bash

Auto-detects every MCP-capable client on the machine, writes the right config for each, installs slash commands, installs the supervisor and bridges (Mac).

If attendee cloned days ago Port numbers changed (8765–8767 → 8865–8867, supervisor added on 8764). They must git pull && bash setup.sh before the session.

Authorize SearchAtlas ~1 min, first run only

Open Claude Code in their workspace, run:

/my-account

A browser tab opens with the SearchAtlas OAuth consent screen. Approve once — token refresh is automatic from here.

The auth probe in each wizard accepts five SA registration shapes (URL match, searchatlas, search atlas, search-atlas, search_atlas) so claude.ai-connector users work alongside claude mcp add users.

1 · Intro 5 min

  • Why this exists. Agencies want SA workflows runnable from Claude Code without typing slash commands. Members get the result without the syntax tax.
  • What they leave with. A working toolkit, 3 web wizards (Onboard, Build, Rebuild), 20+ slash commands, send-integrations for Slack / Discord / email / Circle.
  • Architecture in one breath. 4 local services (supervisor 8764 always-on, three wizard bridges that idle-shutdown). Browser → supervisor → wakes a bridge → bridge spawns claude -p → Claude calls SA MCP. Same flow Mac and Windows.

2 · Install live 7 min

  • Attendees paste the one-liner from the pre-flight section and clone + setup in parallel.
  • Support team in DMs catching install failures.
  • Most common failure: claude CLI not on PATH → tell them to install Claude Code first, restart terminal, re-run.
  • Second most common: port collision on 8865–8867 → run lsof -nP -iTCP:<port> -sTCP:LISTEN to find the offending process.

While they wait, run the diagnostic:

for P in 8764 8865 8866 8867; do
  printf "  %s -> " "$P"
  curl -s --max-time 2 http://localhost:$P/api/health || echo " (no response)"
done

Four JSON blobs with claude_available:true and searchatlas_mcp_configured:true = healthy.

3 · Onboard-client wizard 12 min

End-to-end demo: domain → brand voice → knowledge drop → services → fire.

  • Heartbeat: browser pings /api/heartbeat every 60s — keeps bridges alive while the tab is open.
  • Idle-shutdown: 5 min after the tab closes, bridges exit. Supervisor wakes them on next click.
  • Tab-close safe: Claude subprocess is detached. Closing the tab mid-onboard doesn't kill the run — files land in clients/<slug>/ regardless.
  • First click is ~10 s warm-up. Subsequent clicks are instant. Tell attendees to expect this.
Autoplay demo — internal asset, runs locally on the operator's machine. Open it in a second tab before the block starts so the screen-share is ready.

What to say

  1. Open welcome.html in a second tab. Click Onboard a client. Note the supervisor wake (~10 s).
  2. Paste a demo domain (we'll provide one). Walk through brand voice and knowledge drop steps — flag that the knowledge drop is what beats a plain website scrape.
  3. Pick a service playbook. Hit fire. Show how the chat narration mirrors what Claude is doing.

4 · Build-website wizard 12 min

11-step wizard, ends with a real Website Studio-hosted URL.

  • Each wizard step is a small, single-purpose claude -p subprocess. The Claude CLI here has permission only to call SA MCP tools — narrow blast radius.
  • The audit log (/tmp/amm-website-build-audit.log) records every tool call. Open it in a tab — proves the wizard is calling SA for real.
  • Namespace-tolerant: works with both mcp__searchatlas__* and mcp__claude_ai_Search_Atlas__* installs.

"Looks scary, isn't" talking points

  • "Will this drain my laptop?" One always-on ~15 MB Python process. Everything else idle-shutdowns after 5 min.
  • "What if I close the tab mid-build?" The build keeps running. Files land in clients/<slug>/. Reopen welcome.html later to see status.
  • "Does it cost API credits?" Slash commands run through their Claude Code session — uses their Claude subscription. SA MCP calls use their SA account quotas.
  • "What if I don't trust local servers?" All four bind to 127.0.0.1 only — nothing exposed beyond the machine. Source in tools/ if they want to audit.

One-shot reset script

TOOLKIT="$HOME/.searchatlas/toolkit"
UID_NUM=$(id -u)
for N in supervisor command-center website-build website-rebuild; do
  launchctl bootout "gui/$UID_NUM/com.searchatlas.amm-$N" 2>/dev/null
done
for P in 8764 8865 8866 8867; do
  lsof -t -iTCP:$P -sTCP:LISTEN 2>/dev/null | xargs kill -9 2>/dev/null
done
sleep 2
for N in supervisor command-center website-build website-rebuild; do
  launchctl bootstrap "gui/$UID_NUM" \
    "$HOME/Library/LaunchAgents/com.searchatlas.amm-$N.plist"
done
